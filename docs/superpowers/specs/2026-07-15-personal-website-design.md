# CC 个人网站设计规格

日期：2026-07-15  
状态：已确认

## 1. 项目概述

为 CC 建立一个面向朋友、同行开发者和偶然访客的个人网站。网站使用 Astro Theme Pure 作为成熟底座，不从 Astro Minimal 重新设计通用博客界面。首版重点是准确呈现 CC、支持长期写作，并通过 GitHub 与 Vercel 实现低维护成本的自动发布。

预期观感是简洁、高级、有趣。网站保留 Astro Theme Pure 的明暗主题和克制布局，主要通过像素头像、真实个人信息、文章内容与语言风格形成辨识度，不进行深度主题重写。

## 2. 目标与非目标

### 首版目标

- 访客进入首页后先认识 CC，再发现文章、项目入口和联系方式。
- 统一承载 AI 笔记、搭建记录、免费资源、工具分享和生活随笔。
- 保留项目与友链入口，允许 CC 以后直接修改代码添加内容。
- 支持全文搜索、RSS、SEO、文章目录、响应式布局和明暗主题。
- 源码推送到 GitHub 后由 Vercel 自动构建和发布。
- 保持与 Astro Theme Pure 上游结构接近，降低维护与升级成本。

### 首版非目标

- 不提供 CMS 或网页管理后台。
- 不提供评论、点赞、访客账户、GitHub 登录或邮箱注册。
- 不单独建立 Docs 内容体系。
- 不为项目建立数据库或动态内容集合。
- 不重写 Astro Theme Pure 的底层主题包。

## 4. 信息架构

顶部导航包含：

- 首页
- 文章
- 项目
- 友链
- 关于
- 搜索
- 主题切换

Astro Theme Pure 的 Docs 导航和示例 Docs 内容不进入首版公开网站，生产构建不发布 `/docs` 路由。将来只有在内容形成独立、连续的手册后，才重新设计并启用 Docs。

### 首页顺序

1. 像素头像、CC、杭州 · 余杭区
2. 固定签名
3. About 摘要与详情入口
4. 最近文章
5. Education
6. Projects 入口；没有真实项目时显示“持续更新中”
7. Skills
8. GitHub、邮箱和 RSS

首页不包含 Certifications、模板获取按钮、主题作者项目、随机示例、GPG 信息、赞助二维码或 Lorem ipsum。

### Skills

- Languages：Python、C、C++、Java、HTML、JavaScript、CSS、SQL
- Frontend：React、Vue
- Backend：Flask、Nginx、Next.js
- Others：公益站

Skills 使用文本标签，不显示主观熟练度或进度条。

### Projects 与 Links

- Projects 首版保留页面和导航入口，显示简洁空状态，不使用虚构项目。
- CC 后续直接修改项目页代码添加卡片，不增加 CMS、数据库或项目内容集合。
- Links 首版保留页面和导航入口；没有真实友链时显示空状态，不保留模板作者的链接数据。

## 5. 统一文章系统

所有文章型内容统一存放在 `src/content/blog/`，使用 Markdown 或 MDX。搭建教程与系统性知识仍属于博客，不单独存入 Docs。

首批组织标签为：

- AI 探索
- 搭建记录
- 工具资源
- 生活随笔

一篇文章可以拥有多个标签。文章统一参与标签浏览、全文搜索、RSS、归档和首页最近文章展示。首版允许没有正式文章；此时文章列表显示真实空状态，不保留主题示例文章。

## 6. 视觉与交互

- 沿用 Astro Theme Pure 的整体视觉、排版、响应式组件和交互模式。
- 支持浅色、深色和跟随系统偏好。
- 头像是首页的主要个性化视觉，不额外增加高成本动画或复杂艺术布局。
- 保留清晰的键盘焦点、语义标题层级和图片替代文本。
- 邮箱提供可用的联系入口，并做基础客户端混淆以降低简单爬虫收集风险；不将混淆描述为安全保护。

## 7. 技术架构

- 基底：导入 `cworld1/astro-theme-pure` 的 `main` 分支，并在项目 README 记录导入时的准确上游 commit，保证后续可追溯
- 框架：Astro 6 系列，由模板当前依赖锁定
- 主题集成：`astro-pure`
- 本地 Node：v24.16.0
- 包管理器：沿用模板的 Bun 与 `bun.lock`；本机目前未安装 Bun，在安装项目依赖前补齐
- 内容：Astro Content Collections、Markdown、MDX
- 搜索：Pagefind，由主题现有集成生成
- 部署：Vercel Hobby，用于个人、非商业网站
- 版本管理：Git 与 GitHub

GitHub 仓库的生产分支统一命名为 `main`，初始化阶段将当前本地 `master` 分支改名。Astro 的 `site` 配置从 `SITE_URL` 环境变量读取，本地使用 `http://localhost:4321`，Vercel 项目创建后使用实际生产域名；任何环境都不得保留主题作者的 `astro-pure.js.org`。

不配置 GitHub Actions 部署工作流。Vercel Git 集成负责预览和生产部署。Astro Theme Pure 项目骨架建立后，按已获授权运行 `codegraph init -i` 创建本地代码索引。

## 8. 数据与发布流程

1. CC 修改 Markdown、配置或页面代码。
2. 本地运行检查和正式构建。
3. 变更提交并推送到 GitHub。
4. 非生产分支由 Vercel 创建 Preview Deployment。
5. `main` 分支构建成功后更新 Production Deployment。
6. 构建失败时不替换上一个正常生产版本。

CMS 不参与首版流程。未来若手工编辑文件成为负担，再单独设计 CMS 接入。

## 9. 隐私、外部服务与错误处理

- 禁用模板中指向作者服务的 Waline 配置，避免评论、浏览量或反应数据发送到第三方实例。
- 禁用模板的远程随机语录接口，首页只显示 CC 指定的固定签名。
- 删除模板作者的邮箱、社交账号、项目、友链、赞助、GPG 和二维码数据。
- 必需头像缺失、内容 schema 错误、TypeScript 错误或 Astro 构建错误必须使构建失败。
- Projects、Links 或 Blog 无内容时渲染明确空状态，而不是抛错或显示示例内容。
- 外部 GitHub 链接和邮箱入口在发布前进行人工点击验证。

## 10. 验证与验收

实现完成后至少执行：

- Bun 依赖安装
- Astro/主题检查
- 正式生产构建
- 本地生产预览冒烟测试

人工验收覆盖：

- 桌面与手机布局
- 浅色、深色、跟随系统三种主题行为
- 首页、文章、项目、友链、关于和搜索导航
- 标签、全文搜索、RSS 和文章目录
- GitHub 与邮箱链接
- 空文章、空项目和空友链状态
- 头像、CC、地区、签名、Education 和 Skills 的准确性
- 全站不存在 Astro Theme Pure 作者个人信息、示例项目、示例文章或 Lorem ipsum

## 11. 后续阶段

第二阶段可单独设计评论、点赞和访客账户。目标登录方式是 GitHub OAuth 与邮箱注册，但具体认证服务、数据库、邮件验证、反滥用和隐私方案不属于首版。

CMS 也是独立的未来增强项，不在首版中预留运行时依赖。若未来启用 Docs，也需要基于实际内容规模单独确认信息架构。
